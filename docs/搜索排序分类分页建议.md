# 搜索、排序、分类、分页建议

本文记录当前一轮关于主页交互的分析和建议。

## 排序现状

当前代码支持：

```text
time   按时间
stars  按点赞
views  按浏览
hot    按热度
```

实际 DB 观察：

```text
views         几乎都有值，但不一定持续维护
star_count    约 333,036 帖非零
trace_count   约 336,961 帖非零
comment_count 约 376,413 帖非零
hot           只有约 1,700 帖非零
```

## 建议排序

建议主页只保留：

```text
按时间
按点赞
按讨论
```

可选保留：

```text
按相关
```

### 取消按浏览

理由：

- 浏览数不一定长期维护
- 用户容易误解为稳定热度
- 浏览数很容易被曝光位置影响

### 保留按点赞

理由：

- 含义清楚
- 用户理解成本低
- 虽然点赞数整体不算高，但排序结果可解释

### 不单独做按蹲蹲

理由：

- 和点赞语义接近
- 单独展示会让排序栏变复杂

但蹲蹲应继续在帖子底部指标展示。

### 取消原始按热度

理由：

- `hot` 字段不透明
- 非零数据太少
- 当前排序效果不稳定

如果要保留“综合热度”，建议自己定义公式，并在文档中说明：

```text
score = comment_count * 3
      + star_count * 2
      + trace_count * 1.5
      + log(views + 1) * 0.5
```

更稳妥的名字：

```text
按讨论
```

公式：

```text
comment_count desc, trace_count desc, star_count desc, create_time desc
```

## 分类建议

主页分类下拉只展示前 20 类。

理由：

- 当前分类太多，下拉框噪音大
- 小分类可以通过搜索 tag 解决
- 前 20 类已覆盖主要使用场景

后续可合并近义项：

```text
选课互助 + 选课
期末 + 期末考试 + 考试
大创 + 小创
保研 + 考研
```

但合并会改变用户对原始 tag 的感知，建议第二阶段再做。

## 分页设置建议

建议主页增加一个轻量设置弹层，不做完整设置页。

选项：

```text
每页 10
每页 20
每页 50
每页 100
自定义
```

不建议真正“无限制”。

理由：

- 后端当前最大 limit 会限制到 200
- 前端一次渲染大量帖子会卡
- 对 50 万帖子来说“无限制”没有实际意义

如果保留这个概念，建议 UI 写成：

```text
最多 200
```

自定义范围：

```text
1 - 200
```

保存方式：

```text
localStorage
```

不需要后端存用户设置。

## 推荐改动顺序

1. 主页排序栏改成：时间 / 点赞 / 讨论。
2. 分类接口或前端只显示前 20。
3. 增加设置弹层，保存 `pageSize` 到 localStorage。
4. 后续再考虑相关性排序和自定义综合热度。
