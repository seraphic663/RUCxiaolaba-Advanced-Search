## 项目目前有哪些爬取方式

### Railway 日常生产主线

| 方式 | 源请求 | 做什么 | 是否直接写完整详情 | 当前状态 |
|---|---|---|---|---|
| `discover-latest` | `lists?page=N` | 发现新帖、写 `list_only`、入队 | 否 | 每 30 分钟调度；当天正常运行 |
| `discover-active` | `lists2?page=N` | 发现新回复、评论数变化和活跃缺失帖 | 否 | 每 30 分钟调度；评论增加为 priority 0 |
| `trickle-fill` | `info?id=ID` | 从队列补正文、评论和回复 | 是 | 每 10 分钟尝试，每轮最多 12 个 |
| `plan-gaps` | 通常本地统计；缺结束 ID 时调用一次 `lists?page=1` | 把低 ID 密度区间登记为待抽样范围 | 否 | 每 6 小时调度，不等于逐 ID 扫描 |
| `probe-gaps` | `info?id=ID` | 小样本验证缺口内某 ID 是否真实存在，命中后入队 | 否 | 每日预算为 0，当前实际上不打源请求 |

日常数据流是：

```text
lists / lists2
  -> posts(list_only) + crawler_queue(pending)
  -> trickle-fill
  -> posts(full) + comments + FTS / Bigram / Symbol 索引
```

### Admin 指定内容现爬

管理员可以先从上游 `search`、`lists` 或 `lists2` 预览候选，勾选最多 10 个帖子后选择：

| 方案 | 行为 |
|---|---|
| `smart` | 本地缺失、仅列表状态或上游评论数增加时，立刻请求详情并保存；否则跳过 |
| `force` | 无论本地状态，立刻重新请求详情并保存 |
| `queue` | 不立刻请求详情，只加入 priority -10 的人工优先队列 |

“现爬”在创建任务后由后台线程马上开始，第一个帖子不会等 scheduler 的 10 分钟周期；多选后其余帖子按 8–14 秒间隔串行处理。成功写入时同时更新主库评论和搜索索引。候选预览保存 10 分钟，不把预览内容当正式帖子写进主库。

人工额度现在完全独立：每天 20 次预览页和 10 次详情，不扣自动调度的 `new_list_calls`、`active_list_calls`、`detail_calls`，也不等待 11/14/17/20/23 阶梯释放。它仍使用同一 cookie、同一个上游 session 和全局 pause；一旦上游真实限流，人工与自动任务会一起停到次日。

线上已验证：自动 `new_list_calls` 已到 56 时，Admin 用 `search=食堂` 仍成功返回 10 个候选，人工预览计数从 3 增至 4，自动 new-list 计数保持 56。`search` 的正确上游参数是 `search`，不是 `keyword`。

## API 额度怎么算

### 自动主额度

默认每天最多：

| 类型 | 日上限 |
|---|---:|
| `lists` 新帖列表 | 80 |
| `lists2` 活跃列表 | 160 |
| `info` 详情 | 450 |
| 缺口探测 | 0 |
| 合计 | 690 |

按北京时间阶梯累计释放：

| 时间 | 累计比例 | 主额度累计上限 |
|---|---:|---:|
| 11:00 | 20% | 138 |
| 14:00 | 35% | 241 |
| 17:00 | 50% | 345 |
| 20:00 | 70% | 483 |
| 23:00 | 100% | 690 |

“主要在 11 点和 23 点爬”并不准确。实际释放点有 11、14、17、20、23 点；各任务仍按 30 分钟/10 分钟周期检查。之所以日志容易集中在这些整点附近，是因为新释放的份额通常很快被用完，之后周期检查只会打印额度不足并跳过。23:00 只是释放最后 30%，不是一天唯一的大爬窗口。

最近 14 天如果真实触发过 `rate_limited`，主额度会按最近触顶时实际源调用量的 80% 自适应缩小。人工和自动请求都计入 quota history 的真实 `source_calls`，但人工计数不挤占 690 次主额度。

### 当天观测

20:47 快照的保守预留计数为：new-list 56、active-list 112、detail 268、probe 0；人工预览 4、人工详情 0；主额度自适应比例为 1.0，全局无暂停。

scheduler 会在子进程开始前按最大页数或 `limit` 预留，计数可能略高于实际完成请求。它这样做是为避免并发/崩溃时超额，不能把 quota 文件直接当“成功请求数”。

### 本地额度耗尽与上游真实限流不是一回事

- “对应源 API 额度已用完”：本地软件在发请求前拒绝，本次没有证明上游 session 已限流。
- `rate_limited:*`：请求已经到上游，上游返回“今天刷得太久”等限制文本；此时全部爬取暂停到次日北京时间 00:05，且仍要等待新的阶梯释放。
- `cookie_expired`：登录会话失效，需要合法更新 cookie，不是提高预算能解决。

本次 Admin 报错属于第一种。修复后配置上界是 690 次自动主额度 + 30 次人工额外额度 = 720 次，但真实上游 session 仍是同一个；30 次是小而明确的人工缓冲，不是规避上游限制。

## `limit=12` 的效率

`CRAWLER_TRICKLE_LIMIT_CAP=12` 是每轮最多处理 12 个详情，不是每日只爬 12 个。详情间隔为 8–14 秒，另有网络、SQLite、FTS 和旁路索引写入时间。当天观测中，12 个详情通常耗时约 3.5–4 分钟。

从 11:02 到约 20:29 的日志汇总：

| 项目 | 结果 |
|---|---:|
| 新帖列表预留 | 56 |
| 新帖列表实际完成页 | 52 |
| 活跃列表预留 | 112 |
| 活跃列表实际完成页 | 107，另有 1 次错误请求 |
| 详情选中 | 232 |
| 详情成功写入 | 226 |
| 详情 miss | 6 |
| 上游 rate limit | 0 |

详情写入成功率是 226/232 = 97.4%，miss 为 2.6%。列表加详情的已知实际请求约 392 次，对 400 次保守预留约为 98.0%；预留未完全使用主要来自提前停止或错误。这个结果说明 `limit=12` 的请求执行利用率不差。

但“写入成功”不等于“每次都新增了大量正文/评论”。当前日志的 `queued` 是候选观察次数，在去重/更新前累加，也不能当作唯一新增队列项。要真正评价内容收益，还应增加以下指标：详情前后 `crawl_status`、新增评论行数、正文从空到非空、因数据无变化而重复写入的次数。当前证据只能证明请求大多成功落库，不能证明每个请求的净新增价值都高。

理论上若没有列表扫描阻塞，10 分钟一轮、每轮 12 个，上限是每小时 72 个详情。实际 scheduler 串行运行；例如 20:04–20:22 的两个发现任务占了约 18 分钟，这期间详情任务不能并行启动，所以真实吞吐更低。保持串行是为了 SQLite 写入和 session 请求强度可控，当前不建议为追吞吐直接并发化。

## 能否说“记录基本全了”

只能分两层说：

1. **对已经发现并写入主库的帖子**：98.73% 已有详情，1.27% 仅列表；“主要剩详情补全”成立。
2. **对上游历史全集**：不能证明全了。`lists` / `lists2` 存在窗口、重复页和深度限制；`probe-gaps` 当前预算为 0；历史逐 ID 扫描没有作为日常任务持续跑。删除帖、窗口外帖子和未抽样 ID 缺口都可能无法由现有总数证明。

因此网站适合表述为“已覆盖绝大多数已发现记录，最新列表通常在一个发现周期内追上，详情和历史缺口仍在补”，不应宣称“全量无遗漏”。

## 当前最值得继续改的地方

1. **补收益指标**：在 `crawler/service.py` 和 `storage/post_writer.py` 记录每次详情是 `list_only -> full`、新增多少评论、还是无变化刷新；测试放在 `tests/test_crawler_service.py`。这是判断 450 次详情额度如何分配的基础。
2. **审计 2,624 个 missing 队列项和 32 个未 pending 的 list-only**：先只读按来源、状态、错误和年龄分组，再决定重试或清理，不应直接批量删除。
3. **重新评估 p10/p20 排序**：p10 积压 8,862 时，新出现的 p20 会长期排在后面。可以考虑在不削弱 p0 的前提下给“近期 p20”保留每轮少量槽位；改动前应先用新增评论收益数据模拟每天请求变化。
4. **让发现和详情调度更公平**：不并发打上游，但可以限制一次发现占用的连续墙钟时间，避免 18 分钟发现链完全挡住详情。需在 `jobs/scheduler.py` 加调度回归测试。
5. **把队列观察数改名或另报唯一入队数**：当前日志 `queued=555` 容易被误读为新增 555 条，实际包含已有候选的更新观察。

上述建议都不需要更换身份、轮换 cookie 或绕过限流。任何强度调整都必须保留次日暂停、阶梯释放和用户本人使用小程序的额度缓冲。
